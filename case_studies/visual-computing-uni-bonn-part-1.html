Visual Computing - Uni Bonn : Part 1

Introduction

As a student assistant employed by the Department of Visual Computing in the Institute of Informatics at the University of Bonn, I was tasked with the maintenance and polish of their website. Their backend ran on vanilla KoaJS and MariaDB, and their frontend was rendered with Vue using hand written components. There were a lot of things to improve upon, and I was given a lot of flexibility in choosing which ones to work on first.

Familiarizing myself with the code base

To understand the code better, my first task was to port the way in which the frontend managed state, from Vuex 4 to Pinia (which is essentially Vuex 5). Almost every table in the backend database had an associated store, which meant carefully merging all the different mutations and actions into a single actions object, and making them available along with the state variables through the mapping functions provided by Pinia. At this point, the frontend codebase still used the Options API, and the different Pinia stores also started off with the same API for clarity. Later on, the entire codebase would be ported to the Composition API, along with the stores. Working through all of this provided a good mental model of how data flowed from the backend to the frontend, and was rendered through the different components.

Organizing the UI

After all the stores were ported, my next task was to replace the handwritten UI components with alternatives from a premade UI library, for which we chose PrimeVue. PrimeTek is a Turkish company which creates and maintains an open-source library of UI components for JavaServer Faces, React, Angular, and Vue. With the concern that there could come a time where we might have to shift our entire codebase to another frontend framework, we decided on this library to make our work easier if that happened. Instead of using PrimeVue's own styles however, we mixed it with TailwindCSS to preserve the design guidelines by Uni Bonn.

In the process, we ended up with a clean hierarchy of UI component folders that catered to the different use cases for the website, alongside an App folder for common components. Naturally, the header and footer lived in their own components, but subcomponents were also needed for their different parts, such as the language switcher, the Login Dialog, and the Menu inside the Navbar. Individual Card components were made for different content types. Tables were listed with DataTables and DataViews, with some lists rendered with Accordions. The end result was a very similar looking, but more polished and maintainable version of the website.

Next Steps

Since the backend was created manually, there were a lot of bugs and loopholes resulting in stale data and errors when submitting forms. Therefore, we decided to switch the entire backend itself to a headless cms, and chose Strapi CMS for its popularity. This would mean creating a Strapi instance, setting up all the different content types, and remapping the data fetching to this new backend via the REST API. Such a huge switch took over a year to fully implement without any gaps, and therefore deserves its own part of the case study covering all the different aspects.
