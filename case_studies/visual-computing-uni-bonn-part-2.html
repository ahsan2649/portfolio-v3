Visual Computing - Uni Bonn : Part 2

Introduction

As a student assistant employed by the Department of Visual Computing in the Institute of Informatics at the University of Bonn, I was tasked with the maintenance and polish of their website. In a few months, I helped port the state management to Pinia, and replace vanilla Vue components with PrimeVue, and by this time I was very familiar with the codebase. With that, our next challenge was to implement a headless CMS so that the backend troubles would be minimized.

How a Headless CMS works

Unlike traditional content-management systems like Wordpress, Drupal, and Joomla, which provide a way to create the frontend along with some styles; headless content-management-systems only provide the backend handling. The user creates all the content, but the frontend, or "head" so to say, is managed elsewhere. This way, we wouldn't have to worry about querying the database ourselves for every single operation. On top of this, Strapi provides a plugin for internationalization, where the same content can be queried in different languages, whereas our current backend used duplicated columns for English and German. For us, this means creating the content types and relations that resembled our current backend as much as possible, then swapping the API calls in our frontend to this new backend, and finally remapping the UI to match the new content schema. Alongside this, we also needed a mechanism to transfer all the data from the old backend to this new one, all the while preserving the data while matching it to the new content fields.

Creating the Strapi Instance

Our old backend had 20 tables, a handful of which were pivot tables, and one for sessions. However for the content types in Strapi, we had to create 40. A few of these were because some new content types were added in between switching to the new backend. A quarter of them were geared towards handling our own sessions, users, and permissions. A bunch of content in the old backend also had a type associated with it, which was a single string column. For the new backend, we extracted them into their own types with a relation.

Extra Customizations

Since authentication on the website was split between Local users and LDAP users, we needed to implement our own User Management system instead of the Users and Permissions plugin provided by Strapi. Alongside this, the old backend provided a mechanism where certain course materials could only be accessed by a dedicated local user, therefore we iterated on this to create a full Role-Based-Access-Control system, where individual entries or entire collections could be restricted to certain users or roles. We also created a content type for routes and middleware, so that the frontend can create the entire route tree from the backend, along with the frontend middleware that would run for each route.

Porting Data to the New Backend

For a seamless transition to the new backend, the data that was in the old backend needed to be made available in Strapi. This meant querying the old backend, splitting entries into two languages based on duplicate columns, and sending it to Strapi via the REST API. In order to accomplish this, we used Golang to create a "porting script". The script would utilize and ORM to query the old backend, and for all of the old content types available, it would create the equivalent content in Strapi and send the data via an HTTP Client.

Querying Strapi from the Frontend

As discussed in the previous part, the state management was ported from Vuex to Pinia by this point. Since data fetching is synchronous, however, this means we were inefficiently querying and storing the data, and should have used a server-state manager instead. So in the process of querying the data from the frontend, we also switched most of the data fetching in the Pinia stores to a new library - Tanstack Vue Query. All the queries to the Strapi REST API were now done through Vue Query, except for authentication calls, which were still managed in Pinia.

Conclusion

Porting a vanilla backend to a headless CMS takes a lot of time depending on the scope. This involves copious amounts of trial-and-error testing, a lot of discussion about what the shape of the content should be, and building it up alongside frontend-mapping and data porting. What started in August of 2024 ended up finishing in March of 2026, when the new backend finally went live. Functionally it was the same, but under the hood there were a lot of changes, distinguishable only by the new UI components developed in between for rendering data and displaying forms. Throughout the process I learned an amazing amount of things. REST APIs, middlewares, and backends finally clicked for me, and the approach we now have towards handling data inspired my next portfolio website project, dubbed Portfolio V3.
